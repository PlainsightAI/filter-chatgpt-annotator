## Why LangChain?

The previous architecture was directly coupled to the OpenAI Python SDK. Adding a second provider would have required a custom abstraction layer, per-provider request/response adapters, and a parallel test matrix. LangChain already does all of that work — `init_chat_model` returns a uniform `BaseChatModel`, multimodal `HumanMessage` content is normalized across providers, and `with_structured_output` picks the best JSON-enforcement mechanism for each (tool-calling for OpenAI / Anthropic / Gemini, JSON-mode for Ollama).

## Provider routing

The string passed to `init_chat_model` selects which `langchain-<provider>` integration to load:

| `chattag_model` prefix | Package      | Credential env var       |
| ---------------------- | ------------ | ------------------------ |
| `openai:`              | langchain-openai        | `OPENAI_API_KEY`         |
| `google_genai:`        | langchain-google-genai  | `GOOGLE_API_KEY`         |
| `anthropic:`           | langchain-anthropic     | `ANTHROPIC_API_KEY`      |
| `ollama:`              | langchain-ollama        | `OLLAMA_HOST`            |
| any other LangChain provider | install separately | per provider          |

## Lifecycle

```
setup()      → validate config → build LangChain chain → load prompt
process()    → for each frame: BGR→base64 → HumanMessage(text + image_url)
                              → chain.invoke() → annotations + usage
                              → attach to frame.data["meta"]["chattag"]
                              → optionally append to labels.jsonl + save image
shutdown()   → from labels.jsonl: binary datasets, balanced variant,
                                  COCO multilabel (if schema has >1 label)
```

## What is NOT in the architecture

- No retry layer in this filter (LangChain exposes `.with_retry()` if needed; not wired up by default).
- No tracing (LangSmith is opt-in via env vars; no code path here).
- No response cache.
- No streaming (one-shot per frame; doesn't help here).
